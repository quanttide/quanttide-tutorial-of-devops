# 语义化版本

本文约定量潮科技版本号的规范和使用方式。

## 基本规则

1. alpha 版本：`x.y.z-alpha.n`（n≥1），本地开发版本
2. beta 版本：`x.y.z-beta.n`（n≥1），云端测试版本
3. rc 版本：`x.y.z-rc.n`（n≥1），内部评审版本
4. release 版本：`x.y.z`，正式发布版本

## 正式版本与预发布版本

正式版本面向项目的外部使用者，必须有规范的 release note。要求在仓库根目录维护 `CHANGELOG.md`，格式遵循 Keep a Changelog 规范。

预发布版本作为开发者内部研发流程的阶段性标记：

- **alpha**：本地开发版本。增加一系列特性时阶段性标记，不要求可以在云端使用或本地跑通。
- **beta**：云端测试版本。整个项目可以跑通，一轮测试标记一次版本。
- **rc**：内部评审版本。与正式版的唯一区别是版本号，实际特性应与 release note 一致。

预发布版本不应被其他项目使用，除非特殊情况。

## 基本原则

语义化版本适用于所有需要发布版本的场景，包括客户端版本、库版本、文档版本等。

### 0.0.1 版本号的分界线

0.0.1 是项目的初始版本，项目已创建，初步内容就位。

### 0.1.0 版本号的分界线

0.1.0 是 MVP 版本，核心内容完整、骨架已定，可投入实际使用。在此之前采用 0.0.x 小版本快速迭代原型。

### 1.0.0 版本号的分界线

1.0.0 版本要求 API 保持稳定，核心功能趋于稳定。此阶段可开始大规模投入宣传和生态建设。

## 非公开版本（0.Y.Z）

中版本号定义特性，小版本号约定 API。所有 0.x 版本默认互不兼容，必须指定版本号使用。

## 正则表达式

正式版：

```
^refs/tags/(?:0|[1-9]\d*)\.(?:0|[1-9]\d*)\.(?:0|[1-9]\d*)$
```

含预发布：

```
^refs/tags/(?:0|[1-9]\d*)\.(?:0|[1-9]\d*)\.(?:0|[1-9]\d*)(?:-(?:alpha|beta|rc)(?:\.(?:0|[1-9]\d*))?)?$
```
