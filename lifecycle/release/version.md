# 语义化版本

本文约定量潮科技的版本号规范。

## 私有版本

私有版本面向团队内部使用，版本号仅用于沟通迭代进度，不做兼容性承诺。版本范围为 `0.Y.Z`。

- **0.0.1**：项目初始版本，项目存在的基本标记。
- **0.0.x**（minor = 0）：特例阶段，连"特性/API"的区分都没有确立，纯粹是项目刚创建、骨架刚搭好。
- **0.1.0**：MVP 里程碑，核心内容完整、可投入实际使用。在此之前以 0.0.x 快速迭代原型。
- **0.Y.Z（Y ≥ 1）**：常规迭代。semver 约定"中版本号定义特性，小版本号约定 API"。但由于是私有阶段，实际升级节奏由项目自行决定。
- **1.0.0**：公开版本起点。API 趋于稳定，对外发布。

## 预发布版本

无论私有还是公开，都可以用 prerelease 后缀标记研发阶段。版本号加上 prerelease 后缀，表示"这个版本还差最后一步才能发布"。

- **alpha**：格式 `x.y.z-alpha.n`。本地开发版本，增加一系列特性时阶段性标记，不要求跑通。
- **beta**：格式 `x.y.z-beta.n`。云端测试版本，整个项目可以跑通，一轮测试标记一次。
- **rc**：格式 `x.y.z-rc.n`。内部评审版本，与正式版的唯一区别是版本号，特性应与正式版一致。

同一版本的 prerelease 序号递增：`-rc.1` → `-rc.2`（发现问题修复），最后去掉后缀发正式版。预发布版本不应被其他项目直接依赖，除非特殊情况。

## 场景

### 代码项目

```text
0.0.1          项目初始化
0.0.2          第一个功能跑通
0.1.0-alpha.1  开始开发 MVP 特性
0.1.0-beta.1   MVP 云端测试
0.1.0-rc.1     内部评审
0.1.0          MVP 发布，内部可用
0.2.0          新增特性（私有，不兼容没问题）
1.0.0-rc.1     准备开放
1.0.0          正式公开（严格 semver）
1.1.0          向下兼容新功能
2.0.0          不兼容变更
```

### 文档项目

文档项目没有 API 兼容性问题，版本号反映内容成熟度：

```text
0.0.1   骨架就位，目录结构确定
0.1.0   核心章节写完，可指导实践
0.2.0   补充更多章节，内容趋于完备
1.0.0   内容完备，对外发布
```
