# 计划

## 核心理念：规划即管理预期

计划阶段回答三个问题：

- **要做什么** — 当前版本的交付范围
- **做了什么** — 已经完成的工作
- **还有什么没做** — 剩余的工作量

规划的核心产物是一份与代码同仓的规划文件。它伴随代码一起维护，每次提交前清理已完成条目，保持规划整洁。

规划不是一次性工作，而是伴随每次迭代持续更新：

```text
开发前  → 查看进度           了解当前版本还差什么
开发中  → 标记完成           实时记录进展
提交前  → 验证规划文件格式    确保格式合规
       → 清理已完成条目      保持规划整洁
       → git commit         规划与代码一同提交
开发后  → 查看进度           回顾完成情况，规划下个版本
```

## 规划文件

规划文件位于每个 scope 的根目录，采用结构化格式管理版本条目：每个版本标题下列出该版本的变更分类（新增、修复等），每项工作标记为已完成或未完成。

格式说明和操作命令详见[规划文件](roadmap.md)。

## 常见问题

**scope 是什么？**

scope 是量潮项目中可独立规划、构建和发布的组件。例如 `cli`、`web`、`api` 各是一个 scope，每个 scope 有独立的规划文件。

**能否用项目管理工具替代规划文件？**

规划文件与代码同仓，确保规划与实现同步。推荐二者结合使用：项目管理工具（如飞书多维表格）用于跨团队协作和长期规划，代码仓内的规划文件用于版本级粒度规划。
