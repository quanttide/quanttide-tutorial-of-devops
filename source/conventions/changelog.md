# CHANGELOG

## 核心理念：面向人的变更记录

CHANGELOG 记录每个版本的变化。它面向的是阅读代码的人（协作者、Reviewer、下游用户），不是机器。

好的 CHANGELOG 回答一个问题：**这个版本和我有关吗？**

- Added：新增了什么功能
- Changed：改了什么东西
- Fixed：修了什么 bug
- Removed：删掉了什么

不必罗列每个 commit，而是筛选对使用者有意义的变更。重构、格式化、依赖升级——如果对外部没有影响，就不写。

## 格式

遵循 Keep a Changelog 规范。

### 版本条目

```markdown
## [0.2.0] - 2026-07-04
```

格式：`## [版本号] - YYYY-MM-DD`。预发布版本同样标记日期：

```markdown
## [0.2.0-rc.1] - 2026-07-03
```

### 变更分类

按顺序排列：

```markdown
## [0.2.0] - 2026-07-04

### Added
- 用户认证模块

### Changed
- 重构 API 路由，性能提升 30%

### Fixed
- 修复并发场景下的数据竞争

### Removed
- 废弃的 v1 接口
```

分类不是每版都用，没有就不写。

### 条目粒度

每条变更一句话描述，必要时补充上下文：

```markdown
### Added
- 用户认证模块，支持邮箱和微信登录
- 管理后台操作日志，记录增删改查

### Fixed
- 修复定时任务在跨天场景下触发两次的问题（#142）
```
